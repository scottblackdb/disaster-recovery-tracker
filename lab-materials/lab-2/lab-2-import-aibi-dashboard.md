# Lab 2: Import the FEMA Claims AI/BI Dashboard

**Goal:** Import a pre-built **Databricks AI/BI dashboard** into your workspace so you can analyze the claims data that the Disaster Recovery Tracker app is writing into Lakebase and that has been synced into the Lakehouse.

**Dashboard file:** [`./aibi_dashboard/fema_claims_overview.lvdash.json`](./aibi_dashboard/fema_claims_overview.lvdash.json)

---

## Before you start

- Your instructor has already synced the source data from **Lakebase → Lakehouse** into `fema_claims_workshop_catalog.public.*`. You do **not** need to set this up yourself.
- You should have completed **Lab 1** and submitted at least one claim through the app so there is data to look at.
- You will need access to a **SQL warehouse** in the workspace (any size is fine).

---

## Part A — Download the dashboard file

1. Open the workshop repository in your browser:
   [https://github.com/scottblackdb/disaster-recovery-tracker](https://github.com/scottblackdb/disaster-recovery-tracker)
2. Navigate to `lab-materials/lab-2/aibi_dashboard/`.
3. Open `fema_claims_overview.lvdash.json` and click **Download raw file** (or right-click **Save As…**) to save it locally.

> The `.lvdash.json` extension tells Databricks this is an AI/BI (Lakeview) dashboard definition.

---

## Part B — Import the dashboard into the workspace

1. In the Databricks workspace left nav, click **Workspace**.
2. Navigate to your **Home** folder (or any folder you have write access to).
3. Click the **kebab menu (⋮)** at the top right of the folder view, then choose **Import**.
4. In the **Import** dialog:
   - **Import from:** **File**
   - Drag in or browse to the `fema_claims_overview.lvdash.json` you downloaded.
5. Click **Import**.
6. When the import completes, click the new **`fema_claims_overview`** dashboard to open it.

---

## Part C — Attach a SQL warehouse and run the dashboard

1. With the dashboard open, look at the top-right corner for the **warehouse selector**.
2. Pick any available **SQL warehouse** (a Serverless warehouse will start in a few seconds).
3. The dashboard will run all of its queries against the synced tables in `fema_claims_workshop_catalog.public.*` and render the visuals.

You should see tiles such as:

| Tile | What it shows |
|------|----------------|
| Total claims | Overall claim count from `claims`. |
| Claims by FEMA category | `claims` joined to `fema_categories`. |
| Claims over time | Submissions trended by date. |
| Documents per claim | Counts from `documents`. |
| Status changes | Activity from `claim_status_history`. |
| Total Claims: Actual vs AI Forecast | Historical claim counts plus a 30-day forecast generated by `ai_forecast`. |
| Claim Amounts: Actual vs AI Forecast | Historical estimated cost plus a 30-day forecast generated by `ai_forecast`. |

---

## Part D — Publish and explore

1. Click **Publish** in the top right of the dashboard so others (and you, in view mode) can use it.
2. Try the following to get a feel for AI/BI:
   - Click any visual to **cross-filter** the rest of the dashboard.
   - Use the **Genie** / **Ask a question** entry point (top of the dashboard) to ask a natural-language question such as *"How many claims were submitted this week?"* or *"Which county has the most claims?"*.
   - Click **Edit** to see the SQL behind any tile — every visual is backed by a query against `fema_claims_workshop_catalog.public.*`.

## Part E — Explore the AI Forecast charts

Scroll to the **AI Forecast (next 30 days)** section near the bottom of the dashboard. You'll see two line charts:

- **Total Claims: Actual vs AI Forecast** — daily claim counts plus a forward-looking forecast.
- **Claim Amounts: Actual vs AI Forecast** — daily total estimated cost plus a forward-looking forecast.

Each chart plots four series on the same axis: the historical **actual**, the **forecast** point estimate, and the **upper / lower bound** of the prediction interval (so you can see how confident the model is).

### How is this possible without writing any ML code?

These tiles are powered by Databricks' built-in [`ai_forecast`](https://docs.databricks.com/aws/en/sql/language-manual/functions/ai_forecast) function. It is a regular **SQL table-valued function** — you call it the same way you would call `EXPLODE` or any other SQL function, and Databricks automatically picks an appropriate forecasting model, fits it to your historical data, and returns the predicted values.

Click **Edit** on either forecast tile to see the SQL. The core of the query is just this:

```sql
SELECT * FROM AI_FORECAST(
  TABLE(daily),                              -- your historical data
  horizon => DATE_ADD(CURRENT_DATE(), 30),   -- how far into the future
  time_col => 'submission_date',             -- the time column
  value_col => 'claim_count'                 -- the metric to forecast
)
```

The result is a table with `claim_count_forecast`, `claim_count_upper`, and `claim_count_lower` columns that the dashboard renders alongside the actuals.

### Why this matters

- **No data scientist required.** Any user who can write a `SELECT` statement can add a forecast to a dashboard. There is no model to train, no notebook to run, and no separate ML pipeline to maintain.
- **No new infrastructure.** `ai_forecast` runs on the same SQL warehouse the rest of the dashboard uses. Forecasts refresh automatically every time the dashboard refreshes.
- **Governed by Unity Catalog.** The function reads the same governed tables as your other queries, so forecasts respect the same row/column-level security and lineage as the rest of your data.
- **Works on any time series.** Swap `claim_count` for revenue, ticket volume, sensor readings, or any other numeric metric — and `ai_forecast` will produce a forecast.

In short: **forecasting in Databricks is just SQL.** If you can build a chart, you can build a forecast.

## Part F — Watch live data flow end-to-end

1. Leave the dashboard open in one browser tab.
2. In another tab, open the Disaster Recovery Tracker app and submit a new claim (you can reuse the steps from **Lab 1.1**).
3. Wait ~30 seconds, then click the **Refresh** button at the top of the dashboard.
4. The new claim should appear in the totals and trend tiles.

This is the full loop: the app writes to **Lakebase (OLTP)**, Lakebase **synced tables** replicate the rows into the **Lakehouse**, and the **AI/BI dashboard** queries the Lakehouse — all governed by Unity Catalog.

## Congratulations — you've imported and used an AI/BI dashboard backed by Lakebase!

## Need help?

- **`TABLE_OR_VIEW_NOT_FOUND` errors on the tiles:** The instructor's Lakebase → Lakehouse sync hasn't finished (or hasn't been run). Let the instructor know.
- **Tiles stuck on "Running":** Make sure a SQL warehouse is selected in the top right and that it is in the **Running** state.
- **You don't see your new claim:** Click the dashboard's **Refresh** button. Synced tables typically replicate within seconds, but the dashboard caches results until refreshed.
- **AI Forecast tiles show an error:** `ai_forecast` needs at least ~10 days of historical claims to fit a model. If the workshop catalog is brand new, submit more claims (or wait for more days of data) and refresh the dashboard.
